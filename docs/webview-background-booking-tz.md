# ТЗ: WebView-авторезерв билетов БЖД и перевод сценария в фон

## 1. Цель

Довести авторезерв билетов БЖД до стабильного режима, при котором приложение:

- мониторит наличие мест в фоне;
- при появлении нужного поезда и класса запускает WebView-сценарий бронирования;
- проходит сайт как реальный браузер, включая JS-проверку Verification;
- выбирает нужный поезд, вагон/класс, заполняет данные пассажира;
- оформляет заказ или останавливается в dry-run режиме;
- отправляет статусы и результат в Telegram;
- сохраняет подробные логи для диагностики.

Главная архитектурная идея: не обходить Verification через OkHttp, а выполнять бронирование в настоящем браузерном контексте Android WebView.

## 2. Что уже реализовано

### 2.1. WebView-сценарий бронирования

Реализован `BookingWebViewActivity`, который открывает сайт БЖД в WebView и управляется state machine.

Основной флоу:

1. Открыть route URL.
2. Дождаться страницы маршрута.
3. Найти нужный поезд.
4. Нажать `Выбрать места`.
5. Дождаться `/ru/order/places/`.
6. Найти нужный вагон/класс, например `Вагон №01 (2П)`.
7. Выбрать вагон.
8. Найти кнопку `Ввести данные пассажиров`.
9. При необходимости пройти логин.
10. Заполнить пассажира.
11. Нажать `ОФОРМИТЬ ЗАКАЗ`.
12. Дождаться корзины/успеха.
13. Отправить Telegram-уведомление.

### 2.2. State machine

Реализованы состояния:

- `LOAD_ROUTE`
- `WAIT_ROUTE`
- `SELECT_TRAIN`
- `WAIT_PLACES`
- `SELECT_CARRIAGE`
- `ENTER_PASSENGERS`
- `LOGIN_IF_NEEDED`
- `RESELECT_AFTER_LOGIN`
- `FILL_PASSENGER`
- `SUBMIT_ORDER`
- `WAIT_SUCCESS`
- `DONE`
- `ERROR`

Для состояний есть:

- timeout;
- retry;
- проверка URL;
- проверка DOM;
- подробное логирование результата JS;
- обработка Verification;
- обработка неожиданных редиректов на главную.

### 2.3. DOM-селекторы и действия

Используются ключевые селекторы:

- поезд: `.sch-table__row[data-train-number]`;
- форма выбора мест: `form.js-sch-item-form[action*='/order/places']`;
- кнопка выбора мест: `.btn-index`;
- вагон: `.pl-accord__acc-link`;
- форма перехода к пассажирам: `form.carriage-cost__form`;
- кнопка пассажиров: `a.btn-index-2`;
- логин: `form#form-auth input[name='login']`;
- пароль: `form#form-auth input[name='password']`;
- submit логина: `form#form-auth input[name='dologin']`;
- поля пассажира: `last_name_1`, `first_name_1`, `middle_name_1`, `document_type_1`, `document_number_1`;
- согласие: `input[name='agreement']`;
- финальная кнопка: `form#contact-info_form button[type='submit']`.

### 2.4. Исправленные проблемы

Исправлены найденные проблемы:

- после логина страница перезапускалась, и кнопка выбора мест не нажималась повторно;
- после выбора мест сайт мог возвращать на страницу поездов;
- raw `form.submit()` заменён на клики по реальным JS-кнопкам сайта;
- добавлено подробное логирование в БД;
- устранён ложный переход к пассажирам, когда `carriage-cost__form` уже есть в DOM, но вагон ещё не выбран;
- добавлен автоскролл страницы выбора вагонов;
- шаг автоскролла унифицирован: `330px` для выбора вагона и формы пассажиров;
- скролл в самый низ страницы убран;
- логика выбора вагона проверяет, что нужный вагон раскрыт или был реально кликнут.

### 2.5. Логирование

Реализован `BookingLogger`, который пишет диагностику в таблицу логов.

Логируются:

- state transitions;
- URL;
- `PAGE_STARTED` / `PAGE_FINISHED`;
- результат каждого JS-вызова;
- DOM-диагностика;
- список видимых вагонов;
- координаты target-элемента;
- направление и размер скролла;
- причины ожидания;
- ошибки и timeout;
- результат бронирования.

Пароль не должен попадать в логи.

### 2.6. Интеграция с мониторингом

`MonitorService` продолжает отвечать за поиск мест. При нахождении нужного варианта сценарий передаётся в `BookingCoordinator`, который формирует `BookingRequest` и запускает foreground-сервис бронирования.

Реализован текущий поток:

1. `MonitorService` находит подходящий поезд.
2. `BookingCoordinator` создаёт `BookingRequest`.
3. `BookingForegroundService` поднимает foreground notification и wake lock.
4. `BookingForegroundService` запускает `BookingWebViewActivity`.
5. `BookingWebViewActivity` выполняет WebView-сценарий через `BookingStateMachine`.
6. Activity отправляет статусы и финальный результат через local broadcast.
7. Foreground service обновляет уведомление, отправляет Telegram-статусы и освобождает wake lock.
8. `MonitorService` получает результат и планирует повторную бронь только для реального успешного заказа, не для dry-run.

### 2.7. Foreground service для авторезерва

Реализован `BookingForegroundService`.

Сервис:

- запускается из `BookingCoordinator`;
- создаёт foreground notification с высоким приоритетом;
- держит процесс живым во время активного бронирования;
- владеет `PARTIAL_WAKE_LOCK` с safety timeout;
- запускает `BookingWebViewActivity`;
- получает статусы `ACTION_BOOKING_STATUS`;
- получает финальный результат `ACTION_BOOKING_RESULT`;
- обновляет notification текущим state, сообщением и последним URL;
- имеет action-кнопки `Open booking` и `Stop`;
- отправляет Telegram-статусы старта, Verification, успеха, ошибки и отмены;
- освобождает wake lock при `DONE`, `ERROR`, timeout и ручной отмене;
- отменяет уведомление после завершения сессии.

### 2.8. Activity поверх lock screen и ручная остановка

`BookingWebViewActivity` доработан для foreground-сценария:

- `setShowWhenLocked(true)`;
- `setTurnScreenOn(true)`;
- `FLAG_KEEP_SCREEN_ON`;
- отображение текущего state, сообщения и URL;
- прогресс загрузки WebView;
- кнопка `Stop booking`;
- при ручной остановке отправляется результат с ошибкой;
- при закрытии Activity до завершения сценария отправляется результат с ошибкой;
- Activity принимает команду отмены от foreground service.

### 2.9. Dry-run режим

Реализован явный dry-run режим.

Добавлены:

- поле `autoPurchaseDryRun` в `MonitoringRoute`;
- поле `dryRun` в `AutoPurchaseData`;
- переключатель dry-run на экране создания/редактирования мониторинга;
- передача режима в `BookingRequest`;
- остановка state machine перед финальным submit;
- Telegram-сообщение `dry-run completed`;
- запрет планирования повторной брони после успешного dry-run.

Поведение dry-run:

1. Открыть маршрут.
2. Выбрать поезд.
3. Выбрать вагон/класс.
4. Перейти к пассажирам.
5. Заполнить пассажира.
6. Не нажимать `ОФОРМИТЬ ЗАКАЗ`.
7. Завершить сценарий как успешную проверку без реального заказа.

### 2.10. Диагностика в приложении

Улучшена диагностика авторезерва:

- логи `AUTO_PURCHASE` пишутся в БД;
- экран деталей маршрута умеет фильтровать логи по категории `AUTO_PURCHASE`;
- отображаются последние записи за период хранения;
- добавлено копирование видимых логов;
- foreground notification показывает текущий state, сообщение и последний URL;
- Activity показывает текущий state, сообщение и URL.

## 3. Текущее состояние и ограничения

Кодовый foreground-flow реализован и проверен сборкой. После ручной проверки сценарий работает.

Текущий режим не является полностью невидимым WebView в фоне. Это осознанное архитектурное решение: Android WebView должен работать в реальном Activity-контексте, а процесс удерживается foreground service и wake lock.

Полностью фоновый WebView на Android ненадёжен:

- система может остановить JS/timers/network;
- WebView может быть throttled в фоне;
- background activity launch может быть заблокирован;
- OEM battery saver может убить процесс;
- на заблокированном телефоне WebView может работать нестабильно;
- приложение не должно пытаться обходить PIN/биометрию.

Поэтому приложение использует foreground-сценарий:

- foreground service;
- постоянное уведомление;
- Activity поверх lock screen;
- wake lock на время бронирования;
- full-screen/high-priority notification как fallback.

Оставшиеся ограничения:

- Android/OEM может запретить автоматический launch Activity из фона;
- full-screen intent зависит от системных настроек и политики Android;
- пользователь всё равно может увидеть только notification fallback;
- CAPTCHA/ручная проверка требует вмешательства пользователя;
- приложение не должно и не будет обходить PIN/биометрию.

## 5. Архитектура текущего решения

Компоненты:

- `MonitorService` - мониторинг мест, запуск авторезерва и обработка результата.
- `BookingCoordinator` - принимает найденный маршрут и стартует авторезерв.
- `BookingForegroundService` - удерживает foreground-процесс, notification и wake lock.
- `BookingWebViewActivity` - визуальный WebView-сценарий поверх lock screen.
- `BookingStateMachine` - DOM-автоматизация, таймауты, retry.
- `BookingLogger` - БД-логи и диагностика.
- `TelegramNotifier` - статусы и результат.

Текущий flow:

1. `MonitorService` находит нужный поезд.
2. `BookingCoordinator` формирует `BookingRequest`.
3. `BookingForegroundService` стартует foreground notification и wake lock.
4. Сервис запускает `BookingWebViewActivity`.
5. Activity выполняет WebView state machine.
6. Activity отправляет результат обратно.
7. Сервис освобождает wake lock, обновляет notification, отправляет Telegram.
8. `MonitorService` либо продолжает мониторинг, либо планирует повторный авторезерв после реального успешного заказа.
9. При dry-run повторный авторезерв не планируется.

## 6. Риски

- Verification может не пройти без реально активного WebView.
- DOM сайта БЖД может измениться.
- CAPTCHA или ручная проверка остановит автоматизацию.
- Android/OEM может запретить запуск Activity из фона.
- Battery saver может остановить процесс.
- На телефоне с PIN/биометрией нельзя и не нужно пытаться обходить блокировку.
- Невалидные документы остановят форму. Для паспорта РБ нужен формат: две буквы и семь цифр, например `AB1234567`.
